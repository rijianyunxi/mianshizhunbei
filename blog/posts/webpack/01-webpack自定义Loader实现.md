# Webpack 自定义 Loader 实现深度解析

## 一、Loader 的本质

Loader 本质上是一个 **导出函数的 Node.js 模块**。它接收源文件内容（`source`）作为参数，经过转换处理后返回新的内容（`transformedSource`）。

```js
// 最简单的 Loader 结构
module.exports = function(source) {
  // source: 文件原始内容（字符串或 Buffer）
  // 返回转换后的内容
  return transformedSource;
};
```

Webpack 在构建过程中，遇到匹配规则的文件时，会将文件内容依次传入配置的 Loader 链进行处理，最终输出 JavaScript 模块。

### 1.1 Loader 函数签名

```js
/**
 * @param {string|Buffer} source - 源文件内容
 * @param {object} sourceMap - 可选，上一个 Loader 传递的 source map
 * @param {any} meta - 可选，元数据（可在 Loader 间传递任意数据）
 */
module.exports = function(source, sourceMap, meta) {
  // this 指向 Loader Context（由 webpack 注入）
  return source;
};
```

### 1.2 项目中的 txt-loader 实现

来看我们项目中实际的 `txt-loader.js`：

```js
// webpack/loaders/txt-loader.js
function loadTxt(source) {
    console.log("loadTxt is runing.....", source);
    
    return `
        module.exports = '${source}'
    `
}

export default loadTxt;
```

这个 Loader 做了一件事：**把 `.txt` 文件的内容包装成一个 CommonJS 模块**，让 JavaScript 代码可以直接 `import` 文本文件。

在 `webpack.config.js` 中的配置：

```js
module: {
    rules: [
        {
            test: /\.txt$/,
            use: [
                {
                    loader: './loaders/txt-loader.js'
                }
            ]
        }
    ]
}
```

## 二、Loader 的两个阶段：pitch 与 normal

Webpack 处理 Loader 链时，分为两个阶段：

### 2.1 执行顺序图解

假设配置了三个 Loader：

```js
use: ['loader-a', 'loader-b', 'loader-c']
```

执行顺序如下：

```
pitch 阶段（从左到右）：
loader-a.pitch → loader-b.pitch → loader-c.pitch

normal 阶段（从右到左）：
loader-c → loader-b → loader-a
```

**关键结论：normal 阶段是从右到左（从后往前）执行的！**

### 2.2 pitch 函数

每个 Loader 可以导出一个 `pitch` 方法：

```js
module.exports = function(source) {
  return source;
};

// pitch 方法
module.exports.pitch = function(remainingRequest, precedingRequest, data) {
  // remainingRequest: 当前 Loader 之后的请求字符串
  // precedingRequest: 当前 Loader 之前的请求字符串
  // data: 可以在 pitch 和 normal 阶段之间共享数据
  
  // 如果 pitch 返回了值，则跳过后续 Loader 的 pitch 和 normal 阶段
  // 直接进入前面 Loader 的 normal 阶段（熔断机制）
};
```

### 2.3 pitch 的熔断机制

```
正常流程：
pitch-a → pitch-b → pitch-c → normal-c → normal-b → normal-a

pitch-b 返回值时（熔断）：
pitch-a → pitch-b（返回值）→ normal-a
```

**pitch 的实际应用场景：**

- `style-loader` 利用 pitch 熔断，直接返回注入 CSS 的 JS 代码，跳过后续 Loader 对文件的读取
- 实现 Loader 间的数据传递（通过 `data` 对象）

```js
// style-loader 的 pitch 简化版
module.exports.pitch = function(remainingRequest) {
  return `
    var content = require(${JSON.stringify('!!' + remainingRequest)});
    require(${JSON.stringify(require.resolve('./addStyle'))});
    module.exports = content;
  `;
};
```

## 三、Loader Context（this 上下文）

在 Loader 函数中，`this` 指向 Webpack 注入的 **Loader Context**，提供了丰富的 API。

### 3.1 this.query / this.getOptions()

获取 Loader 的配置参数：

```js
// webpack.config.js 中配置参数
{
    loader: './loaders/my-loader.js',
    options: {
        prefix: '/* Generated by my-loader */',
        suffix: '/* End */'
    }
}

// my-loader.js 中获取参数
const { getOptions } = require('loader-utils'); // 旧版方式

module.exports = function(source) {
    // 新版 Webpack 5 推荐方式
    const options = this.getOptions({
        // JSON Schema 校验
        type: 'object',
        properties: {
            prefix: { type: 'string' },
            suffix: { type: 'string' }
        }
    });
    
    return `${options.prefix}\n${source}\n${options.suffix}`;
};
```

### 3.2 this.callback 返回多个值

当需要返回多个值（如 source + sourceMap）时，使用 `this.callback`：

```js
module.exports = function(source, sourceMap) {
    // this.callback(err, content, sourceMap, meta)
    this.callback(
        null,           // 错误（null 表示无错误）
        source,         // 转换后的内容
        sourceMap,      // source map（可选）
        {}              // meta 数据（可选）
    );
    // 注意：使用 this.callback 后，不能再 return 值
};
```

### 3.3 异步 Loader

```js
module.exports = function(source) {
    // 声明这是一个异步 Loader
    const callback = this.async();
    
    // 执行异步操作
    someAsyncOperation(source, (err, result) => {
        if (err) {
            callback(err);
            return;
        }
        callback(null, result);
    });
};
```

### 3.4 其他常用 API

```js
module.exports = function(source) {
    // 添加文件依赖（文件变化时触发重新编译）
    this.addDependency('/path/to/some/file');
    
    // 获取当前文件路径
    console.log(this.resourcePath);
    
    // 获取 webpack 配置的 context
    console.log(this.rootContext);
    
    // 标记为可缓存（默认 true）
    this.cacheable(true);
    
    // 发出警告
    this.emitWarning(new Error('some warning'));
    
    // 发出错误（不中断构建）
    this.emitError(new Error('some error'));
    
    return source;
};
```

## 四、使用 loader-runner 独立测试 Loader

`loader-runner` 是 Webpack 内部使用的 Loader 执行器，可以脱离 Webpack 单独测试 Loader：

```bash
npm install loader-runner --save-dev
```

```js
// test-loader.js
const { runLoaders } = require('loader-runner');
const path = require('path');
const fs = require('fs');

runLoaders(
    {
        resource: path.resolve(__dirname, './test.txt'), // 要处理的文件
        loaders: [
            {
                loader: path.resolve(__dirname, './loaders/txt-loader.js'),
                options: {
                    // Loader 配置
                }
            }
        ],
        context: {
            // 模拟 Loader Context
            emitFile: () => {}
        },
        readResource: fs.readFile.bind(fs) // 文件读取方法
    },
    (err, result) => {
        if (err) {
            console.error('Loader 执行出错:', err);
            return;
        }
        console.log('转换结果:', result.result);
        console.log('依赖文件:', result.fileDependencies);
    }
);
```

## 五、完整的自定义 Loader 实战

### 5.1 实现一个 banner-loader（给每个文件添加版权注释）

```js
// loaders/banner-loader.js
const { validate } = require('schema-utils');

// 定义 options 的 JSON Schema
const schema = {
    type: 'object',
    properties: {
        author: { type: 'string' },
        license: { type: 'string' }
    },
    additionalProperties: false
};

module.exports = function(source) {
    // 1. 获取并校验配置
    const options = this.getOptions(schema);
    
    // 2. 构建 banner 注释
    const banner = `
/**
 * @author ${options.author || 'Unknown'}
 * @license ${options.license || 'MIT'}
 * @date ${new Date().toISOString()}
 */
`.trim();
    
    // 3. 返回处理后的内容
    return `${banner}\n${source}`;
};

// 标记为可缓存
module.exports.cacheable = true;
```

### 5.2 实现一个 i18n-loader（国际化替换）

```js
// loaders/i18n-loader.js
module.exports = function(source) {
    const options = this.getOptions();
    const translations = options.translations || {};
    
    // 替换 __('key') 形式的翻译占位符
    const result = source.replace(/__\('([^']+)'\)/g, (match, key) => {
        return JSON.stringify(translations[key] || key);
    });
    
    return result;
};
```

## 六、面试高频考点

### Q1：Loader 的执行顺序是什么？

**答：**

1. **normal 阶段从右到左（从后往前）执行**，即配置数组中最后一个 Loader 最先处理文件
2. **pitch 阶段从左到右执行**，与 normal 阶段相反
3. 完整顺序：`pitch-a → pitch-b → pitch-c → normal-c → normal-b → normal-a`

**为什么是从右到左？** 这是函数组合（compose）的设计，类似 `a(b(c(source)))`，最内层的 `c` 先执行。

### Q2：pitch 有什么用？

**答：pitch 的核心作用有三个：**

1. **熔断机制**：pitch 返回值时，跳过后续所有 Loader 的 pitch 和 normal 阶段，直接进入前面 Loader 的 normal 阶段。`style-loader` 就利用这个机制避免重复读取文件。

2. **数据传递**：通过 `data` 对象在 pitch 和 normal 阶段之间共享数据：
   ```js
   module.exports.pitch = function(remainingRequest, precedingRequest, data) {
       data.value = 'hello'; // pitch 阶段写入
   };
   module.exports = function(source) {
       console.log(this.data.value); // normal 阶段读取 → 'hello'
       return source;
   };
   ```

3. **前置处理**：在文件内容被读取之前执行一些逻辑（如权限检查、缓存判断）

### Q3：同步 Loader 和异步 Loader 的区别？

| 特性 | 同步 Loader | 异步 Loader |
|------|------------|------------|
| 返回方式 | `return` 或 `this.callback()` | `this.async()` 获取 callback |
| 适用场景 | 简单字符串转换 | 文件 I/O、网络请求 |
| 阻塞 | 阻塞 Webpack 构建线程 | 不阻塞 |

### Q4：Loader 和 Plugin 的区别？

| 维度 | Loader | Plugin |
|------|--------|--------|
| 作用 | 文件转换（非 JS → JS） | 扩展 Webpack 功能 |
| 运行时机 | 模块加载时 | 整个构建生命周期 |
| 实现方式 | 导出函数 | 导出带 `apply` 方法的类 |
| 配置位置 | `module.rules` | `plugins` 数组 |
| 典型例子 | `babel-loader`、`css-loader` | `HtmlWebpackPlugin`、`MiniCssExtractPlugin` |

## 七、总结

自定义 Loader 的核心要点：

1. **本质是函数**：`source => transformedSource`，可同步可异步
2. **执行顺序**：normal 阶段从右到左，pitch 阶段从左到右
3. **pitch 熔断**：返回值时跳过后续 Loader，`style-loader` 的核心机制
4. **this 上下文**：提供 `getOptions`、`callback`、`async`、`addDependency` 等丰富 API
5. **独立测试**：使用 `loader-runner` 脱离 Webpack 测试 Loader 逻辑
