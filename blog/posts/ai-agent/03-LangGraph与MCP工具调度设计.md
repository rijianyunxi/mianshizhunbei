# LangGraph 与 MCP 工具调度设计

前面一篇写了 Koa 服务入口和基础路由，但如果只停在 Web 框架层，这个项目的价值其实还没展开。

这个 AI Agent 项目真正有意思的部分，在于它不是一个“调用大模型返回字符串”的简单后端，而是把：

- **LangGraph 状态图**
- **MCP 工具协议**
- **动态工具路由**
- **会话持久化**

组合成了一个可扩展的 Agent 执行系统。

本文结合 `agent-koa` 里的真实代码，拆一下它的工具调度链路。

---

## 一、先看这个项目到底解决什么问题

项目定位不是通用聊天，而是一个偏业务场景的智能助手。

在 `smartConstructionAgent.js` 中，系统提示词已经明确了角色：

```js
const AGENT_SYSTEM_PROMPT = [
  '你是“智慧工地 AI 调度助手”。',
  '你的任务：',
  '1. 优先保证施工安全和合规。',
  '2. 输出结构化、可执行的步骤和检查项。',
  '3. 必要时主动调用工具，不要凭空编造现场数据。',
  '4. 默认使用简体中文，表达简洁清晰。',
  '5. 工具调用遵循“能力感知 + 按需最小化”原则：仅在当前已启用工具能直接解决问题时调用对应工具。',
  '6. 若用户请求涉及外部操作（如网页、设备、数据系统），先判断是否存在匹配工具；有则执行，无则明确说明缺失能力并给出替代方案。',
].join('\n')
```

这里已经说明了两个核心目标：

1. **回答不是纯文本生成，而是带工具执行能力的决策输出**
2. **工具不是全量暴露，而是按需最小化暴露**

第二点非常关键，因为 Agent 一旦把所有工具 schema 都塞给模型，很快就会遇到上下文膨胀、选错工具、工具描述噪音太大这些问题。

---

## 二、为什么这里要用 LangGraph，而不是手写一个 while 循环

很多人第一次做 Agent，会写成这种伪代码：

```js
while (还没结束) {
  const response = await llm.invoke(messages)
  if (response 要调用工具) {
    const toolResult = await callTool()
    messages.push(toolResult)
  } else {
    return response
  }
}
```

这样当然能跑，但问题也很明显：

- 状态流转不清晰
- 中途失败不容易恢复
- 很难接入持久化 checkpoint
- 复杂场景下调试成本很高

LangGraph 的价值就在于：

> **把 Agent 执行流程抽象成状态图，而不是散落在代码里的 if / else。**

这样做之后，系统会更像一个“可恢复的工作流引擎”，而不只是一次性函数调用。

---

## 三、Agent 运行时实际上由哪些部分组成

从代码结构看，这个项目的 Agent 运行时主要分成 4 层：

### 1. Web 接入层

也就是 Koa 路由：

- `/agent/chat`
- `/v1/chat/completions`
- `/agent/threads`
- `/admin/mcp/servers`
- `/rpc/mcp/call`

负责接收请求、校验参数、组织上下文、返回流式结果。

### 2. Agent 核心层

`src/agent/smartConstructionAgent.js`

负责：

- 创建模型
- 组织消息
- 选择工具
- 执行 Agent invoke / stream
- 控制最大迭代次数和错误恢复

### 3. 工具层

包括：

- `mcpRegistry.js`
- `toolRouter.js`
- `KeywordSearch.js`

负责：

- 连接 MCP Server
- 拉取工具描述
- 根据用户问题筛选最相关工具
- 对工具做统一封装

### 4. 持久化层

包括：

- `checkpointer.js`
- `conversations.js`

负责：

- LangGraph checkpoint 存储
- 会话列表和消息记录
- thread 级别恢复

---

## 四、MCP Registry：把工具进程抽象成统一注册中心

MCP 在这里不是一个概念摆设，而是真正承担了“工具插件系统”的角色。

看 `mcpRegistry.js`，它做的事情本质上是：

1. 启动 MCP Server 子进程
2. 用 MCP Client 建立连接
3. 拉取工具列表
4. 维护启用/禁用状态
5. 暴露统一的 `callTool` 能力

这一步的意义是：

> **模型不需要关心工具来自哪个进程、哪个协议细节，它只需要面对统一抽象后的工具描述。**

这跟前端里的“组件注册中心”有点像：外部能力很多，但使用层希望拿到统一接口。

---

## 五、为什么还需要 Tool Router

如果 MCP Registry 已经能列出所有工具，为什么还要一个 `toolRouter`？

因为：

> **“能调用所有工具”和“应该暴露哪些工具给模型”是两回事。**

这套系统里，`toolRouter` 的职责不是执行工具，而是做**工具检索与筛选**。

`toolRouter.js` 里有两个明显的设计点：

### 1. 向量检索

它使用了：

- `OllamaEmbeddings`
- 向量相似度匹配

把工具描述转换成 embedding，再根据用户请求召回 TopK 工具。

### 2. 关键词检索兜底

它还保留了 `KeywordSearch`：

- 名称命中权重更高
- 描述次之
- schema / serverId 权重更低

这很实用，因为真实项目里向量检索不是永远可靠：

- 有时 embedding 模型不稳定
- 有时业务词汇非常垂直
- 有时工具名本身就比语义更重要

所以这套设计是 **语义召回 + 规则兜底** 的组合，而不是把希望全押在 embedding 上。

---

## 六、这个项目最值得写进面试里的点：按需最小化暴露工具

这是整套 Agent 里最有工程味道的设计之一。

很多 Agent Demo 都会犯一个问题：

- 一次性把几十个工具全部交给模型
- 每个工具都附带很长 schema
- 最后模型选错工具，或者频繁 hallucination

而这个项目的思路是：

1. 先从全部已启用工具里检索出最相关的一小部分
2. 再把这部分工具封装成 LangChain Tool
3. 最后只把它们交给当前轮模型

这能解决几个很现实的问题：

- 降低 token 开销
- 降低 schema 噪音
- 提高工具选择准确率
- 让新增 MCP 工具时不会线性恶化上下文质量

如果面试官问“你们 Agent 做了什么工程优化”，这一点完全可以重点讲。

---

## 七、LangChain Tool 封装这一层做了什么

在 `smartConstructionAgent.js` 里，MCP 工具不会直接裸奔给模型，而是被包装成 `DynamicStructuredTool`。

这样做的好处是：

- 输入 schema 统一
- 工具描述统一
- 调用返回格式统一
- Agent 层不需要感知工具实现细节

也就是说，MCP 负责“跨进程工具协议”，而 LangChain Tool 负责“模型可消费的工具接口”。

这两层并不重复，而是刚好接在一起。

你可以把它理解成：

- **MCP** 解决“工具怎么接进来”
- **DynamicStructuredTool** 解决“模型怎么调用它”

---

## 八、Agent 不是无限调用工具，而是有熔断策略

真实场景里，Agent 如果没有边界，很容易出现：

- 工具循环调用
- 工具失败后反复尝试
- 一个请求拖很久才结束

所以这套实现里有几个关键限制项：

- `recursionLimit`
- `AGENT_TOOL_MAX_CALLS`
- `AGENT_TOOL_TIMEOUT_MS`

这说明作者不是只关注“功能能跑”，而是考虑了运行时约束。

这种设计特别像后端里的“限流 / 熔断 / 超时保护”：

> Agent 本质上也是一种运行时系统，不是提示词拼装器。

---

## 九、为什么 thread_id 很重要

这套系统里，所有对话不是纯无状态请求，而是围绕 `thread_id` 进行。

`thread_id` 的作用至少有三层：

### 1. 关联 LangGraph Checkpoint

这样 Agent 的中间状态能够恢复。

### 2. 关联会话消息存储

比如你在会话列表里看到的标题、最后一条消息、更新时间，都跟 thread 绑定。

### 3. 支持多轮对话上下文

这意味着接口不只是“单轮问答 API”，而是更接近完整聊天系统。

所以这不是一个“用户发一句，模型回一句”的玩具项目，而是已经具备会话系统雏形。

---

## 十、这一套工具调度链路怎么串起来

把整个流程串起来，其实可以概括成：

```txt
用户请求
→ Koa 路由校验参数
→ 根据 thread_id 组织上下文
→ toolRouter 从 MCP 工具集合里召回 TopK
→ MCP 工具被包装成 DynamicStructuredTool
→ LangGraph 驱动 Agent 推理 / 工具调用循环
→ 工具结果回注到消息流
→ 最终输出文本或流式响应
```

如果要更偏工程一点，还可以继续补上：

```txt
→ 同时写入 conversationStore
→ 同时依赖 SQLite checkpoint 持久化状态
```

## 十一、这部分内容在面试里怎么讲

如果面试官问：**“你这个 AI Agent 后端和普通聊天接口有什么区别？”**

可以这样答：

### 1. 不是单纯的 LLM proxy

它不只是把前端消息转发给模型，而是加了一层 Agent 运行时。

### 2. 有工具系统

通过 MCP Registry 把外部能力注册成可调用工具。

### 3. 有工具路由层

不是把所有工具一次性丢给模型，而是做 TopK 召回，控制上下文噪音。

### 4. 有持久化和线程模型

支持 thread 级别的多轮对话和状态恢复。

### 5. 有流式输出和兼容层

既支持自定义接口，也支持 OpenAI Compatible 风格接口，方便接前端。

这套回答会比“我做了一个 AI 聊天项目”强很多。

因为这部分刚好能把后端“能跑”提升到“能接入前端、能持续维护”。

---

## 总结

这个 `agent-koa` 项目最值得看的，不是 Koa 本身，而是它把 Agent 系统拆成了几层清晰职责：

- Koa 路由负责入口
- LangGraph 负责执行流
- MCP Registry 负责工具接入
- Tool Router 负责工具选择
- SQLite 负责状态持久化

这说明它不是一个 demo 式拼装项目，而是已经开始往**可扩展 Agent 后端架构**的方向走了。

如果你自己也在做 AI Agent 项目，我很建议你重点吸收这两个点：

1. **工具不要全量暴露给模型**
2. **对话状态不要只存在内存里**

这两个决定，基本会直接影响项目后面能不能继续长大。
