# MCP 管理后台与热插拔设计

很多 AI Agent 项目把工具系统做到这一步就停了：

- 工具能注册
- 模型能调用
- demo 能跑

但一旦项目往真实使用场景走，就会马上遇到一个更实际的问题：

> 工具不是一劳永逸写死的，它会新增、下线、替换、调试，甚至临时启停。

这也是为什么 `agent-koa` 里除了 Agent 主链路，还专门做了一层 **MCP 管理接口**。

这一篇就专门讲这块。

---

## 一、为什么工具系统需要“管理后台思维”

如果工具是写死在代码里的，短期当然简单：

```js
const tools = [weatherTool, searchTool, dbTool]
```

但这样的问题也很明显：

- 每加一个工具都要重启服务
- 工具故障时不方便隔离
- 无法快速做 A/B 调试
- 很难做插件式扩展

而 MCP 的优势，本来就是把工具能力从主进程里拆出去。  
既然拆出去了，那下一步自然就会走到：

- 怎么看当前有哪些 MCP Server
- 怎么启用 / 停用某个 Server
- 停用后路由怎么同步更新
- 新 Server 接入后，模型什么时候能感知到它

所以“管理后台”不一定非得先做可视化页面，**先有管理接口就已经是关键的一步**。

---

## 二、这套管理能力的入口在哪里

在 `src/routes/mcpAdmin.js` 里，项目提供了几个核心接口：

- `GET /admin/mcp/servers`
- `POST /admin/mcp/servers/enable`
- `POST /admin/mcp/servers/disable`

光看这三个接口，就已经能看出设计意图：

### 1. 列表接口

查看当前 MCP Server 状态，以及 tool router 的状态。

### 2. enable 接口

允许动态传入：

- `id`
- `command`
- `args`
- `cwd`
- `description`
- `env`

也就是：**不仅能启用已有 server，还能按配置动态接入一个新 server。**

### 3. disable 接口

按 `id` 停掉对应 MCP Server。

这其实已经是一套很像插件平台的最小可用控制面了。

---

## 三、为什么 enable 接口的参数设计很关键

`enableSchema` 里这一组字段非常重要：

```js
const enableSchema = z.object({
  id: z.string().min(1),
  command: z.string().min(1).optional(),
  args: z.array(z.string()).optional(),
  cwd: z.string().optional(),
  description: z.string().optional(),
  env: z.record(z.string(), z.string()).optional(),
  enabled: z.boolean().optional(),
})
```

这意味着 MCP Server 并不是只能从固定配置文件中读取，而是可以：

- 按命令启动
- 带参数启动
- 指定工作目录
- 指定环境变量
- 动态注册说明

本质上，这已经不是“工具列表开关”那么简单，而是：

> **你在运行时动态挂载一个外部工具进程。**

这是非常像插件系统或扩展系统的设计。

---

## 四、MCP Registry 在热插拔里扮演什么角色

热插拔这件事，真正的核心并不在路由，而在 `mcpRegistry`。

因为路由层只是：

- 接参数
- 做校验
- 调 registry
- 返回结果

真正负责生命周期管理的是注册中心。

`mcpRegistry` 至少承担了这些职责：

### 1. 规范化插件配置

比如处理：

- command
- args
- cwd
- env
- enabled

### 2. 创建 MCP Client 和 Transport

通过 stdio 把主服务和 MCP 子进程连起来。

### 3. 维护服务状态

比如：

- 哪些 server 处于 active
- 每个 server 暴露了哪些工具
- 工具的描述与 schema 是什么

### 4. 提供统一调用能力

外层只需要按：

- server_id
- tool_name
- arguments

去调用，不需要知道底层连接细节。

所以热插拔真正成立的前提是：

> **工具生命周期已经被注册中心托管起来了。**

---

## 五、为什么“启停 server”之后还要重建 tool router

这点特别容易被忽略。

很多人会以为：

- MCP Server 启动成功了
- 那模型下一轮自然就知道了

其实不一定。

因为模型真正可见的，不是“某个 server 进程在线”，而是：

- 当前有哪些工具描述
- 这些工具怎么被检索
- 哪些工具会出现在 TopK 结果里

而这些正是 `toolRouter` 管的。

所以当 MCP Server 变化时：

1. 工具集合变了
2. 工具描述索引变了
3. 检索语料库变了
4. TopK 召回结果也会变

因此：

> **热插拔不只是进程级启停，还包括工具索引的同步刷新。**

如果这一步不做，系统就会出现一种很别扭的状态：

- server 明明已经启用了
- 但 Agent 还是找不到它的工具

或者反过来：

- server 明明已经停了
- router 里还残留旧工具描述

这就是为什么管理接口和 tool router 必须联动。

## 六、hot-swap-test-mcp 这个测试 server 有什么意义

在 `src/mcp/hotSwapTestServer.js` 里，项目专门准备了一个测试型 MCP Server。

这个设计非常聪明，因为热插拔这种能力很难只靠脑补验证。

它需要一个：

- 很轻量
- 很稳定
- 结果容易观察
- 不依赖复杂业务环境

的测试 server。

`hot_swap_health`、`hot_swap_counter_next` 这类工具的价值就在于：

- 一眼能看出 server 是否真的启动了
- 一眼能看出是不是同一个进程实例
- 一眼能验证启停后状态是否重置

这种测试 server 在工程里特别有用，因为它让“热插拔是否真的生效”变成可以直接观测的事实。

---

## 七、demoSmartSiteServer 为什么也很重要

`demoSmartSiteServer.js` 则更像业务化演示工具集。

它提供的是：

- 塔吊状态查询
- 到岗统计
- 安全巡检类能力

这说明项目不是为了展示 MCP 而展示 MCP，而是把它落在了具体业务语义里。

换句话说：

- `hotSwapTestServer` 用来测系统能力
- `demoSmartSiteServer` 用来承载业务能力

这两类 server 同时存在，说明作者已经在区分：

- **平台能力验证**
- **业务工具接入**

这个边界分得很清楚。

---

## 八、热插拔这件事为什么对 Agent 项目特别重要

在普通后端里，动态注册插件已经很有价值。  
但在 Agent 场景里，它更重要，原因有三点。

### 1. 工具能力经常变化

今天要接数据库，明天要接搜索，后天还要接工单系统。

### 2. 工具经常不稳定

外部系统宕机、依赖命令失效、账号权限变化，都可能导致工具临时下线。

### 3. Agent 对工具集合非常敏感

工具多一个少一个，模型的行为都可能变化。

因此在 Agent 项目里，工具系统必须是“可运营”的，而不是“写死后不动”的。

---

## 九、这套设计的工程价值在哪里

如果你把这部分抽象一下，会发现它解决的不只是 MCP 本身，而是更大的问题：

### 1. 解耦主服务和外部能力

主服务不需要把所有业务 SDK 都直接依赖进来。

### 2. 降低新增工具成本

新增一个工具，不一定要改 Agent 主进程逻辑。

### 3. 更方便隔离故障

某个 MCP Server 挂了，可以单独下线，而不是整个服务跟着重启。

### 4. 提升调试效率

启一个测试型 server，看看 registry 和 router 是否刷新成功，就能快速定位问题。

这几条都非常像中后台 / 平台型系统的设计思路，不是玩具项目会认真考虑的东西。

---

## 十、如果继续往前走，这里可以怎么升级

现在这套接口已经很好用了，但如果要继续增强，我会优先考虑这些方向。

### 1. 增加状态事件和审计日志

例如记录：

- 谁启用了哪个 server
- 启动参数是什么
- 启停时间点
- 是否成功重建 router

这样更适合团队协作。

### 2. 支持健康检查

不只是“进程启动成功”，而是主动探测：

- server 是否还能响应
- 工具列表是否一致
- 最近一次调用是否异常

### 3. 支持版本化与灰度

未来可能出现：

- 同一个业务工具有 v1 / v2 两个版本
- 想先让部分请求灰度命中某个 server

### 4. 做成真正后台界面

现在是管理接口，下一步可以补一个轻量控制台：

- server 列表
- 在线状态
- 工具数量
- 启停按钮
- 测试调用按钮

这样就从“有管理能力”升级成“可视化管理后台”。

---

## 十一、面试里怎么讲这一块

如果面试官问：**“你们的 Agent 工具系统和普通 function calling 有什么区别？”**

可以这样回答：

### 1. 工具不是静态数组

我们做了 MCP Registry，把工具能力外置成独立 server。

### 2. 支持动态启停

通过管理接口可以在运行时启用或停用某个 MCP Server。

### 3. 工具变化会联动路由层

server 启停后会刷新 tool router，保证工具检索结果和实际能力一致。

### 4. 有专门的热插拔测试 server

可以验证 registry、router、工具状态是否真正同步。

### 5. 这套系统更像插件平台

后端主服务负责调度，外部工具能力以插件方式接入。

这会比“我们支持 function calling”这种泛泛表达强很多。

---

## 总结

`agent-koa` 在 MCP 这块最值得看的，不只是“接了几个工具”，而是它已经开始具备**工具平台化**的雏形：

- 有 registry 管生命周期
- 有 admin route 管启停
- 有 tool router 管检索和暴露
- 有测试型 MCP Server 验证热插拔链路
- 有业务型 MCP Server 承载真实能力

这说明它不是把 MCP 当噱头，而是真的在往“可扩展 Agent 工具系统”这个方向设计。

如果后面你继续补前端控制台，那这一块会更完整。  
但即使只看现在的后端接口，它已经足够写进一个不错的项目专题了。
